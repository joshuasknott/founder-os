# FounderOS Backend Schema (Convex)

## Overview
The FounderOS backend is powered by **Convex**. It acts as a real-time, strictly typed relational database. The architecture is designed around two core principles:
1. **Strict Multi-Tenancy:** Every single query must be scoped to a `workspaceId`. Data bleeding between workspaces is a critical failure.
2. **The Append-Only Ledger:** Agent actions and system history are never mutated or deleted; they are appended to an immutable ledger for perfect auditability.

---

## 1. Core Entity Tables

### `workspaces`
The root boundary for all data. 
- `name` (String): e.g., "FounderOS".
- `logo` (String, Optional): URL or icon string.
- `createdAt` (Number): Epoch timestamp.

### `users`
Human collaborators managed via BetterAuth (or Clerk/NextAuth).
- `externalId` (String): The ID from the auth provider.
- `workspaceId` (Id<"workspaces">): The active workspace.
- `name` (String): Full name.
- `email` (String): Email address.
- `role` (String): "Owner" | "Contributor".
- `avatarUrl` (String, Optional).

### `agents`
The AI workforce roster.
- `workspaceId` (Id<"workspaces">): The workspace they belong to.
- `name` (String): e.g., "Orion", "Cipher".
- `role` (String): e.g., "Chief of Staff", "Vanguard".
- `model` (String): e.g., "gpt-4o", "claude-3-5-sonnet".
- `systemPrompt` (String): The core instructions.
- `status` (String): "Idle" | "Working" | "Offline".
- `clearanceLevel` (Number): 1 (Auto-execute) to 3 (Requires Human Approval).

---

## 2. Intelligence & Knowledge

### `knowledge_nodes`
A polymorphic table replacing traditional nested file systems. Handles both folders and files.
- `workspaceId` (Id<"workspaces">): Scope constraint.
- `parentId` (Id<"knowledge_nodes">, Optional): Null if at root level.
- `type` (String): "Folder" | "File" | "Blueprint".
- `title` (String): Name of the item.
- `content` (String, Optional): Markdown text or document data (for files/blueprints).
- `metadata` (Object, Optional): JSON for tags, author, or codebase extraction data.

---

## 3. Operations & Telemetry

### `approval_queue`
Tasks generated by Level 2/3 Agents waiting in the CEO's Rightnav Inbox.
- `workspaceId` (Id<"workspaces">): Scope constraint.
- `agentId` (Id<"agents">): The agent proposing the action.
- `title` (String): High-level task name (e.g., "Deploy V2 Infrastructure").
- `status` (String): "Pending" | "Approved" | "Denied".
- `proposedPayload` (String): JSON or Code diff of the intended action.
- `justification` (String): The agent's reasoning.
- `createdAt` (Number).

### `event_ledger`
The immutable history of everything executed (Populates the Intelligence Hub History tab).
- `workspaceId` (Id<"workspaces">): Scope constraint.
- `actorType` (String): "Human" | "Agent".
- `actorId` (String): ID of the User or Agent.
- `action` (String): Description of the event (e.g., "Deleted knowledge file").
- `targetResource` (String): What was affected.
- `timestamp` (Number).
